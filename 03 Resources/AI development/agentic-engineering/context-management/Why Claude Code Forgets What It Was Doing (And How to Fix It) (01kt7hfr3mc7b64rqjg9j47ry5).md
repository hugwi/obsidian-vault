Every cloud code session has a finite context window. At some point in a long session, it fills up. When it does, cloud account runs compaction. The conversation history is summarized and the raw message are discarded. What the model knew about your task from the last 50 tool calls is gone. What survives is the summary. The current cloud code MD and whatever you type next. This episode is about designing your setup so that compaction does not matter. You will build two very usable templates, a task state file and a session handoff file 

and understand the cloud MD token discipline that keeps your context budget from being eaten by your own configuration. This is the most architectural episode in the series. I will use diagrams because the concepts are more important than the code. To understand why compaction is a problem, you first need to understand what actually in the context window. Cloud code context window is not just the conversation. It contains several things all competing for the same budget. First conversation story. Every user message, 

every assistant response, every tool called input and the output, all of it accumulates turn by turn in a long coding session with hundreds of file reads and bash commands. Here's how I implement it. The task state file is single most important pattern in this series. The idea is simple. The current task state lives in 

a file, not in a conversation. Cloud writes to the file as it works. When compaction happens, the file is not affected. It was on disk before compaction and it's on disk after. Here is the template four section. Current task done. Next step, an open questions. This is the most important section for continuity. When a new session starts, it wraps this line and begins there. Open questions capture anything unresolved. A deliberate session split is different from a compaction. A compaction happens when the context fails and cloud the code 

takes over. A deliberate split is a choice. You decide the session is complete. You write a handoff file and you start a new session that reads it. Deliberate splits are underused. Most engineers treat them as a failure mode. They are actually a better pattern than relying on compaction to manage continuity. You control what is preserved. You decide what the next session should know. You discard everything that is no longer relevant. Here is the session handoff template. Five sections. roughly 10,000 tokens consumed every single turn. Before any conversation content, before any file reads, before any tool results or a 200 turn session, that is 2 million token turns of cloud AMD content alone. The three principles of the cloud AMD token discipline. First, structural content pays for itself. A file map that helps cloud navigate to the repo. For example, listing the directories and their purposes saves token down streams 

because the cloud does not need to explore the file tree. Put it in behavioral pros rules do not pay for themselves. They add tokens and may or may not be followed. Be selective. Second, cut the wish list. Most cloud AMD files accumulate rules that were added after a bad session and never remove. Always add tests. Never use magic strings. Remember to update change log. These are not instructions Cloud follows reliably. They are tokens spent 

on probabilistic reminders. More the important constraint into the hooks. Remove the rest. Third, use hierarchy. A 200 line project level cloud MD is loaded for every task in that project. A 20 line subdirectory cloud MD is only loaded when cloud is working in subdirectory put context where it's needed not at the top level where it's always below. The before and after for this repo the first draft of the cloud MD was around 250 lines. After applying these principles, keeping all structural 

content, removing behavioral pros, moving constraints to the hook, it came down to 180 lines, roughly 2,500 tokens freed per term across a long session that is meaningful extension of effective context budget. The last concept is when deliberately split a session rather than let compaction manage the continuity. The default instinct when compaction hits is continue to session. Sometimes that's right. Often it is not. Here are four signals that tell you a deliberate split 

is a better choice. First, the task phase changed. You finished implementing a feature and now you are starting the test phase. A new session with a clean context and a short hand of file is more focused than a compacted continuation. Second, the context is contaminated. A long debugging session leaves a lot of dead end reasoning in the context. Even after compaction, the summary preserves some of that noise. Starting fresh with only known facts is cleaner. Third, the cloud AMD changed. If you update the 

project cloud MD significantly during a session, a new session loads the updated version cleanly from the start. A compacted continuation has the old version in the summary. Fourth, you are approaching a critical operation before a database migration, a deployment or a major refactor. Write a handoff file and start a new session. You want maximum context budget for the operation itself, not a depleted budget from a work that preceded it. The operational pattern, it's a right session handoff MD commit or save all in progress.
